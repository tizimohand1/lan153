1. Bruteforcing SSH

Attack Overview: SSH brute force attacks attempt to gain unauthorized access by systematically trying different username/password combinations. This section demonstrates both the attack technique and defensive countermeasures.

Attack Methodology

In this section, we'll use the ncrack tool to perform a brute force attack against the SSH service on UbuntuSRV, then observe how sshguard detects and mitigates the attack.

Tools Used

  • ncrack: Network authentication brute forcing tool
  • sshguard: Intrusion detection and prevention tool for SSH
  • iptables: Linux firewall for blocking malicious traffic

Learning Objectives

2. Demonstrate DOS Attack

CRITICAL WARNING: Do not attempt this section on a personal computer. It will cause serious harm to a machine, resulting in an inoperable state. This is for educational demonstration only in a controlled lab environment.
Attack Overview: A fork bomb is a type of denial-of-service attack that depletes available system resources by creating an exponentially increasing number of processes, ultimately making the system unresponsive.

Understanding Fork Bombs

The fork bomb attack we'll demonstrate uses a bash function that recursively calls itself, rapidly consuming system memory and CPU resources. The attack syntax :(){ :|: & };: creates a function that pipes its output to itself while running in the background.

Monitoring System Resources

1 Launch Resource Monitor: On UbuntuSRV, start htop to monitor system resources
sysadmin@ubuntusrv:~$ htop

This will show real-time CPU and memory usage.

2 Open Second Terminal: Right-click Terminal icon and select "New Window"

Position both terminals side-by-side so you can monitor the attack's impact.

Executing the Fork Bomb

3 Launch Fork Bomb: In the second terminal, execute the attack
sysadmin@ubuntusrv:~$ :(){ :|: & };:
4 Observe System Impact: Watch the htop display for 3-4 minutes
  • CPU usage will spike to nearly 100%
  • Memory usage will rapidly increase
  • Swap memory will also be consumed
  • The system will become increasingly unresponsive
What's Happening: The fork bomb creates multiple copies of itself exponentially, creating a chain reaction that quickly exhausts system resources. Each process forks two new processes, leading to exponential growth: 2, 4, 8, 16, 32, 64... until system limits are reached.

System Recovery

5 Force System Shutdown: When the system becomes unresponsive, use the VM controls

Click on the Ubuntu dropdown menu and select "Power Off"

6 Restart System: Wait for shutdown to complete, then click "Power On"

The system will boot normally as the attack doesn't cause permanent damage.

Mitigation Strategies: In production environments, implement process limits (ulimits), resource quotas, and monitoring systems to detect and prevent such attacks before they can render systems unusable.

3. Destroying the HDD with dd

EXTREME WARNING: Do not attempt this section on a personal computer. This will cause permanent, irreversible damage to storage devices, rendering them completely inoperable. This demonstration is for educational purposes only in a disposable lab environment.
Attack Overview: The dd command can be weaponized to perform destructive disk operations by overwriting critical system areas with zeros or random data, effectively destroying the filesystem and making data unrecoverable.

Understanding the dd Command Attack

The command dd if=/dev/zero of=/dev/sda reads zeros from the special device /dev/zero and writes them directly to the primary storage device /dev/sda, overwriting the master boot record, partition table, and all data.

Setting Up Monitoring

1 Launch UbuntuSRV: Log in as sysadmin with password NDGlabpass123!
2 Start I/O Monitor: Launch iotop to monitor disk activity
sysadmin@ubuntusrv:~$ sudo iotop

This will show real-time disk read/write activity.

3 Open Second Terminal: Right-click Terminal and select "New Window"

Position both terminals to view them simultaneously.

Executing the Destructive Attack

4 Launch Disk Attack: Execute the destructive dd command
sysadmin@ubuntusrv:~$ sudo dd if=/dev/zero of=/dev/sda
5 Observe Disk Activity: Watch the iotop output for 1-3 minutes

You'll see intense write activity as zeros are written to the disk at maximum speed.

Technical Details: The dd command bypasses the filesystem and writes directly to the raw device. This overwrites critical system areas including the boot sector, partition table, and all user data, making the system completely unbootable.

System Destruction and Recovery Attempt

6 Wait for System Crash: Allow the attack to run until the system becomes unresponsive

The system will eventually crash as critical system files are overwritten.

7 Force Shutdown: Use VM controls to power off the system

Select the dropdown menu and choose "Power Off"

8 Attempt Restart: Try to power the system back on

Click "Power On" and observe the boot process.

9 Verify Destruction: Observe the boot failure

The system will display "Operating System not found" or similar error, confirming the attack succeeded in destroying the bootable system.

Attack Success: The dd command has successfully destroyed the system by overwriting critical boot sectors and filesystem structures. In a real environment, this would represent complete data loss requiring restoration from backups.

Defense Strategies

  • Access Controls: Restrict sudo privileges and administrative access
  • File System Protections: Implement immutable flags on critical system files
  • Monitoring: Alert on unusual disk I/O patterns
  • Backup Systems: Maintain offline, verified backups
  • Physical Security: Prevent unauthorized physical access to systems

1.2 Observe the Action of sshguard

Configuring sshguard

1 Edit sshguard Configuration: On UbuntuSRV, modify the configuration for faster response
sysadmin@ubuntusrv:~$ sudo nano /etc/sshguard/sshguard.conf

Change BLOCK_TIME to 60 and DETECTION_TIME to 60. Save with Ctrl+S, exit with Ctrl+X.

Configuration Note: We're using shorter timeouts than production defaults to make the lab demonstration more apparent. In production, longer detection and block times are typically more appropriate.
2 Check Initial Firewall Rules: Examine iptables before starting sshguard
sysadmin@ubuntusrv:~$ sudo iptables -S
3 Start sshguard: Enable the protection service
sysadmin@ubuntusrv:~$ sudo service sshguard start sysadmin@ubuntusrv:~$ service sshguard status
4 Observe Firewall Changes: Check how sshguard modifies iptables
sysadmin@ubuntusrv:~$ sudo iptables -S

Notice the new sshguard chain has been created.

Configuring Automatic Blocking

5 Enable SSH Port Monitoring: Configure iptables to send SSH traffic to sshguard
sysadmin@ubuntusrv:~$ sudo iptables -A INPUT -p tcp -m tcp --dport 22 -j sshguard

Testing the Protection

6 Launch Second Attack: On Kali, run ncrack again
kali@kali$ ncrack -v 172.16.1.10 --user sysadmin -P /usr/share/wordlists/fasttrack.txt -p ssh

This time, the attack should fail due to sshguard blocking.

7 Verify Blocking: On UbuntuSRV, check the updated firewall rules
sysadmin@ubuntusrv:~$ sudo iptables -S

You should see a rule blocking 203.0.113.2 (the Kali machine).
